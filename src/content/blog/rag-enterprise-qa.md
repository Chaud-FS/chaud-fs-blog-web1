## 背景与动机

企业知识库最大的两个痛点是**幻觉**和**知识滞后**。直接把私有文档塞进 prompt，不仅撑爆上下文，还会让模型「编」出不存在的条款。我在 **Enterprise AI Assistant** 里采用经典 RAG 链路：上传 → 切片 → 向量化 → 检索 → 拼接 → 生成，把答案锁死在可溯源的内部资料上。

> 一句话经验：**检索质量决定回答下限，生成模型决定回答上限**，两头都要抓。

## 整体架构

```
文档上传 → 解析(PDF/Word) → 结构感知切片 → Embedding → 向量库(Chroma)
                                                  ↑
用户提问 → 向量检索 Top-K → Cross-Encoder 重排 → 拼接上下文 → LLM → 流式返回
```

后端用 **FastAPI**，编排用 **LangChain**，向量库选本地 **Chroma**（零运维、可随服务打包），前端用 SSE 把 token 一个个推出来。

## 切片：最容易被忽视的环节

切太大 → 噪声多、易超长；切太小 → 语义被腰斩。我的做法：

- **结构感知优先**：PDF 先用 `pdfplumber` 提取带坐标的文本块，按标题层级（H1/H2）切分，保留「章节—小节」父子关系，而不是暴力按字数切。
- **块大小 500–800 字符**，重叠（overlap）取 **15%**，避免句子在边界被切断。
- 每块附带**元信息**：`source_file`、`page`、`section`，回答时原样回传，前端可点击跳到原文。

| 策略 | 优点 | 坑 |
| --- | --- | --- |
| 固定窗口 | 实现快 | 常切坏表格 / 代码 |
| 递归字符 | 较鲁棒 | 仍可能断句 |
| 结构感知 | 质量最高 | 依赖文档结构质量 |

## 检索与重排

纯向量召回（ANN）经常「相关但不精准」。我在召回 Top-20 之后接一层 **Cross-Encoder 重排**：

```python
hits = vector_store.similarity_search(query, k=20)
pairs = [(query, h.page_content) for h in hits]
scores = reranker.predict(pairs)          # cross-encoder 打分
top = [hits[i] for i in argsort(scores)][:4]
context = "\n---\n".join(h.page_content for h in top)
```

这一步对「精准问答」提升最明显，尤其当多个片段都「沾边」时。

## 多轮对话与指代

多轮要处理**上下文窗口**和**指代消解**：

- 维护滚动历史，拼装时优先保留最近 N 轮；
- 用户追问「它指的是什么」时，把上一轮问题一并带入再做检索，避免丢失指代；
- 对长历史做**摘要压缩**，防止 prompt 膨胀推高成本和延迟。

## 流式输出与工程细节

FastAPI 的 `StreamingResponse` + 生成器即可逐 token 返回；前端用 `fetch` + `ReadableStream` 消费。注意：

- 给流式接口加**超时与断流保护**，模型卡住时前端不会永久转圈；
- 回答引用的片段用 `[1]` 上标 + 底部来源列表呈现，满足企业「可审计」诉求。

## 小结

RAG 落地的难点不在「接一个 LLM」，而在**切片策略、向量检索、重排与溯源**这一整条链路。把每一环做扎实，企业级问答才真正可用。
